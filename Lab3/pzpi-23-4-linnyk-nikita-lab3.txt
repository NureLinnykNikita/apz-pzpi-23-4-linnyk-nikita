        Розробити front-end частину програмної системи.

забезпечувати адміністрування системи, тобто мати окремий інтерфейс для звичайних користувачів і окремий — для адміністраторів (відповідно до ролей користувачів вашої системи).

Адміністрування системи повинно забезпечувати: управління користувачами системи, управління даними системи, створення резервних копій налаштувань та даних, експорт та імпорт даних та налаштувань.

Назва інструменту та версія: Anthropic Claude Sonnet 4.6, Claude-Design
Мета використання: проєктування дизайн-системи веб-застосунку, створення окремих елементів користувацького та адміністративного інтерфейсів, опрацювання структури навігації, підтримки локалізації та інтернаціоналізації, а також допомога у формуванні архітектурних рішень front-end частини системи.
Внесок автора: самостійне проєктування архітектури front-end частини системи, реалізація користувацького та адміністративного веб-інтерфейсів, розробка функціональної логіки взаємодії з серверною частиною системи, реалізація підтримки української та англійської мов, адаптація інтерфейсу відповідно до регіональних налаштувань, а також доопрацювання та інтеграція запропонованих інструментом рішень відповідно до вимог проєкту.
3.1 ОПИС ПРИЙНЯТИХ ІНЖЕНЕРНИХ РІШЕНЬ
3.1.1 Архітектурні рішення
3.1.1.1 React SPA з Vite
Веб-клієнт реалізовано як Single Page Application на React 19 із збирачем Vite. Рішення обумовлено потребою у швидкому HMR під час розробки, нативній підтримці TypeScript, а також сумісності зі стеком mobile-клієнта (React Native). Відсутність SSR спрощує розгортання: результатом збірки є набір статичних файлів.
3.1.1.2 TanStack Query v5 для серверного стану
Увесь серверний стан, зокрема списки курсів, статистика та повідомлення, управляється через TanStack Query. Це забезпечує автоматичне кешування, фонові оновлення, дедублікацію запитів і декларативні ключі запитів (queryKey). Інвалідація кешу (invalidateQueries) використовується після мутацій для підтримання актуальності даних без ручного перезавантаження.
3.1.1.3 Zustand для клієнтського стану
Аутентифікаційний стан, зокрема access token і профіль користувача, зберігається в Zustand-сторі. Access token зберігається виключно в памʼяті, а не в localStorage, що знижує ризики, повʼязані з XSS. Refresh token зберігається в httpOnly cookie на стороні сервера.
3.1.1.4 Axios з interceptor для оновлення токенів
Інстанс api (axios) містить interceptor відповіді: при отриманні 401 він автоматично робить запит на /api/refresh, отримує новий access token і повторює оригінальний запит. Введено захист від циклічного виклику через прапорець isRefreshing і чергу refreshQueue. Для logout введено окрему функцію logoutDirect на чистому axios без interceptor, щоб уникнути циклу оновлення: вона включає поточний access token у заголовок вручну.
3.1.1.5 CSS Custom Properties як дизайн-система
Увесь дизайн реалізовано через CSS custom properties (змінні --lb-*). Темна тема є значенням за замовчуванням у :root. Світла тема перевизначає ті самі змінні у [data-theme="light"]. Перемикання відбувається встановленням атрибута на document.documentElement. Тема застосовується синхронно в main.tsx до першого рендеру React, що виключає появу "flash of wrong theme".
3.1.1.6 localStorage-прапорці для персистентності
Чотири прапорці в localStorage управляють персистентними налаштуваннями без залежності від API при кожному завантаженні:
– lb_lang — мова інтерфейсу;
– lb_timezone — часовий пояс;
– lb_theme — тема (dark/light);
– lb_session — наявність активної сесії, що запобігає появі 401 у консолі при оновленні сторінки без cookie.
3.1.1.7 Intl API для форматування
Форматування дат і чисел реалізовано через браузерні Intl.DateTimeFormat та Intl.NumberFormat. Рішення обрано замість date-fns через нативну підтримку часових поясів за допомогою параметра timeZone в Intl. Усі функції форматування зосереджені у файлі web/src/i18n/locale.ts і автоматично враховують поточний timezone з localStorage.
3.1.1.8 Prisma з @map для перейменування без міграцій
При перейменуванні поля dailyGoalMinutes на dailyGoalExercises у схемі Prisma збережено @map("daily_goal_minutes"). Це дозволило перейменувати поле на рівні Prisma Client без зміни стовпця в PostgreSQL: достатньо виконати prisma generate без db push і без ризику втрати даних.
3.1.1.9 Захист від самодеградації прав адміністратора
На рівні сервера при виконанні умови req.user.userId === id під час зміни ролі або видалення повертається відповідь 403. На рівні UI власний рядок адміністратора в таблиці блокує Select ролі та кнопку видалення, показуючи бейдж "YOU".
3.1.1.10 Рішення щодо безпеки
Рішення щодо безпеки реалізовано на рівні клієнтської і серверної частин системи. До основних рішень належать:
– ролевий контроль доступу: чотири ролі (admin, contentManager, moderator, user) з окремими наборами дозволів у БД;
– зберігання access token у памʼяті (Zustand), а refresh token — в httpOnly cookie;
– валідація tokenVersion при кожному запиті, що дає змогу анулювати всі токени при зміні пароля або logout;
– використання компонента RequireRole на фронтенді та middleware на сервері;
– визначення senderId виключно з req.user.userId (JWT), а не з тіла запиту.

3.2.1 Загальна структура
Загальну структуру взаємодії компонентів системи наведено на рисунку 2.1.
  

Рисунок 2.1 — Загальна структура компонентів системи
3.2.1.1 Потік аутентифікації при завантаженні
Потік аутентифікації при завантаженні застосунку має таку послідовність:
– main.tsx синхронно викликає applyTheme(getTheme()) і встановлює data-theme на <html>;
– App.tsx монтує AuthRestorer, після чого перевіряється localStorage.lb_session;
– якщо прапорець наявний, виконується POST /api/refresh через plain axios без interceptor, отримується новий access token, викликаються setAccessToken(), GET /users/me та setAuth(token, user), після чого встановлюється lb_session;
– якщо прапорець відсутній, одразу виконується setReady(true) без мережевого запиту та без 401 у консолі;
– React Router визначає маршрут, а RequireAuth перевіряє isAuthenticated.
3.2.1.2 Потік зміни налаштувань (Settings)
Зміна налаштувань користувача виконується за такою схемою:
– користувач змінює timezone у Select;
– setLocalTz(tz) спричиняє миттєвий повторний рендер LocalePreview;
– setTimezone(tz) оновлює localStorage.lb_timezone;
– settingsMutation.mutate({ timezone: tz }) надсилає PATCH /api/users/me/settings;
– після успішної відповіді qc.invalidateQueries(['settings']) запускає background refetch у TanStack Query;
– useEffect([settings?.timezone]) синхронізує setLocalTz з даними сервера.
3.2.1.3 Взаємодія компонентів UI
Взаємодія компонентів користувацького інтерфейсу організована таким чином:
– Pages підписуються на TanStack Query через хуки useQuery та useMutation;
– Card, Button, Input, Toggle, Select є безстановими UI-компонентами й приймають стиль через змінні --lb-*;
– Sidebar читає useAuthStore для відображення імені користувача та виконання logout;
– AdminLayout перевіряє роль через useAuthStore і формує навігацію через ROLE_SECTIONS map;
– Icons реалізовано як функціональні SVG-компоненти, що приймають size, stroke та style;
– InitialAvatar генерує кольоровий аватар з першої літери імені.

	3.3 UML-ДІАГРАМИ СИСТЕМИ
3.3.1 UML-діаграми прецедентів
UML-діаграма прецедентів надана у додатку Б(Б.1).
UML-діаграма прецедентів відображає функціональні можливості системи LangBang Web та ролі користувачів, які з ними взаємодіють. На діаграмі виділено основних акторів: гостя, авторизованого користувача, адміністратора, менеджера контенту та модератора. Гість має доступ до реєстрації, входу та відновлення пароля, тоді як авторизований користувач може працювати з навчальним функціоналом, профілем, налаштуваннями та рейтингом. Адміністратор, менеджер контенту й модератор мають розширені права відповідно до своїх ролей.
3.3.2 UML-діаграми компонентів
UML-діаграма компонентів надана у додатку Б(Б.2).
UML-діаграма компонентів показує загальну архітектуру системи та взаємодію між її основними частинами. Клієнтська частина складається з точки входу, маршрутизації, сторінок, layout-компонентів, UI-компонентів, сховища Zustand, TanStack Query, API-шару, модуля локалізації та темізації. Серверна частина містить маршрути, контролери, сервіси, middleware, Prisma Client і взаємодіє з базою даних Neon PostgreSQL. Діаграма демонструє, як запити з браузера проходять через Axios до Express-сервера, обробляються бізнес-логікою і передаються до бази даних.
3.3.3 UML-діаграми пакетів
UML-діаграма пакетів надана у додатку Б(Б.3).
UML-діаграма пакетів описує логічну структуру проєкту та розподіл файлів за основними модулями. У клієнтській частині виділено пакети сторінок, компонентів, API, стану, локалізації, темізації та типів. Серверна частина поділена на маршрути, контролери, сервіси, middleware, помилки та Prisma-схему. Також окремо показано структуру mobile-клієнта. Діаграма допомагає зрозуміти організацію коду, залежності між пакетами та відповідальність кожного модуля.
3.3.4 UML-діаграми взаємодії
UML-діаграма взаємодії надана у додатку Б(Б.4).
UML-діаграма взаємодії відображає основні сценарії роботи системи у вигляді послідовності дій між браузером, Zustand Store, TanStack Query, Axios Client, Express Server і PostgreSQL. На діаграмі показано процес початкового завантаження застосунку, відновлення сесії, автентифікації, виконання вправ, зміни налаштувань, перемикання теми, експорту даних адміністратором і виходу з системи. Вона демонструє, як клієнтська частина, сервер і база даних взаємодіють під час виконання ключових функцій платформи.
4 ВИСНОВКИ
У результаті аналізу встановлено, що прийняті інженерні рішення забезпечують модульність, підтримуваність і можливість подальшого розширення системи. Використання React SPA з Vite спрощує розробку інтерфейсу та розгортання застосунку.
Розділення стану між TanStack Query та Zustand підвищує передбачуваність роботи клієнтської частини й зменшує дублювання логіки. TanStack Query використовується для серверних даних, а Zustand — для локального стану автентифікації.
Реалізовані механізми безпеки, зокрема зберігання access token у памʼяті, httpOnly cookie для refresh token, перевірка tokenVersion і ролевий контроль доступу, формують багаторівневий захист системи.
CSS Custom Properties, темізація, localStorage-прапорці та Intl API покращують користувацький досвід і забезпечують гнучке налаштування інтерфейсу.
Застосування Prisma та PostgreSQL забезпечує надійну й типізовану роботу з базою даних. Загалом обрані рішення відповідають функціональним вимогам і створюють основу для подальшого розвитку навчальної платформи.
