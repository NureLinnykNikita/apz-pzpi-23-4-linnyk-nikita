3.1 Загальні відомості про систему
Назва системи: Factorio 
Розробник: Wube Software 
Призначення системи: Програмна система імітаційного моделювання промислових і логістичних процесів у реальному часі. Цільовою аудиторією є користувачі, що проєктують складні автоматизовані системи, та дослідники алгоритмів маршрутизації. Основні функції включають детерміновану симуляцію мільйонів взаємодіючих об'єктів (сутностей), обчислення поведінки логічних схем (комбінаторів), розрахунок гідродинамічних мереж та підтримку синхронізованої багатокористувацької взаємодії.


3.2 Архітектурний стиль системи
Система базується на парадигмі проєктування, орієнтованого на дані (Data-Oriented Design, DOD) у поєднанні з архітектурним шаблоном «Сутність-Компонент-Система» (Entity-Component-System, ECS). Для мережевої взаємодії використовується гібридна клієнт-серверна архітектура з жорсткою детермінованою синхронізацією кроків (Deterministic Lockstep).
Обґрунтування вибору архітектури: Вибір зумовлений екстремальними вимогами до продуктивності та пропускної здатності процесора. Класичне об'єктно-орієнтоване програмування призводить до сильної фрагментації пам'яті. Використання парадигми DOD гарантує послідовне розміщення однотипних даних у пам'яті. Це максимізує ефективність використання кешу процесора (рівнів L1/L2) та критично знижує кількість зупинок конвеєра процесора (processor stalls) і хибних передбачень розгалужень (branch mispredictions). Архітектура Lockstep дозволяє мінімізувати обсяг мережевого трафіку: між вузлами передаються виключно дії користувачів, тоді як повний стан масивної симуляції паралельно і детерміновано розраховується на кожному клієнті.


3.3 Основні компоненти системи
Опис підсистем:
* Інтерфейс користувача: Підсистема візуалізації, побудована на базі власного графічного рушія. Забезпечує рендеринг спрайтів із використанням апаратного прискорення та обробку подій вводу.
* Серверна частина: Автономний сервер (Headless server), який виконує роль маршрутизатора команд, арбітра з'єднань та генератора ідентифікаторів тактів симуляції.
* Рівень бізнес-логіки: Ядро симуляції. Виконує періодичне оновлення стану системи (60 тактів на секунду). Поділене на ізольовані модулі: модуль маршрутизації логістичних дронів, гідродинамічний модуль (розрахунок тиску та потоків рідин, наприклад, при переробленні нафти), модуль арифметичних та вирішальних комбінаторів.
* Рівень зберігання даних: Підсистема серіалізації. Забезпечує збереження поточного стану оперативної пам'яті у стиснені двійкові файли зі збереженням посилальної цілісності між сутностями.
* Допоміжні сервіси: Підсистема інтеграції модифікацій (Modding API) на базі ізольованого середовища виконання мови Lua.


3.4 Взаємодія компонентів
Взаємодія базується на головному циклі симуляції. Підсистема вводу реєструє дії користувача і передає їх на сервер. Сервер групує команди від усіх клієнтів за тактами симуляції (Ticks) і розсилає їх назад. Отримавши пакет команд, ядро симуляції клієнта виконує такт обчислення. Усі модулі бізнес-логіки обробляють свої компоненти суворо детерміновано. Після завершення обчислень підсистема візуалізації зчитує координати об'єктів (тільки для читання) для побудови кадру.
  

Рисунок 1.1 - UML-діаграма послідовності


Опис діаграми: UML-діаграма послідовності демонструє життєвий цикл обробки команди у мережевому режимі. Показано асинхронну відправку команд на сервер, їх буферизацію та подальший синхронний детермінований розрахунок логіки ядра на клієнтах без передачі загального стану світу.


3.5 Масштабованість та надійність
* Механізми балансування навантаження: На рівні симуляції застосовується багатопотоковість для обчислення незалежних графів системи. Наприклад, фізично не з'єднані між собою гідродинамічні труби або окремі електричні мережі розраховуються у паралельних потоках операційної системи.
* Відмовостійкість: Реалізовано механізм виявлення десинхронізації (Desync detection). Система періодично обчислює контрольну суму стану пам'яті ядра симуляції. У разі розбіжностей ініціюється процедура відновлення – примусове завантаження еталонного стану від сервера.
* Кешування: Дані структуровані за принципом «Масив структур» (Structure of Arrays), що унеможливлює фрагментацію та гарантує максимальне потрапляння даних до кешу (cache hit rate) під час ітерацій в ігровому циклі.
* Реплікація: Замість класичної реплікації стану об'єктів (State Replication) використовується реплікація вводу (Input Replication). Кожен вузол системи є повноцінною реплікою логіки.
* Контейнеризація: Підтримується розгортання серверної частини (Headless) у середовищах Docker для автоматизації управління безперервними сесіями.

Архітектурні принципи та приклади програмного коду, що демонструють підхід проєктування, орієнтованого на дані, наведено у Додатку В.


4 ВИСНОВКИ
У ході виконання роботи було досліджено, що система імітаційного моделювання Factorio досягає високої продуктивності завдяки використанню Data-Oriented Design у поєднанні з архітектурою Entity-Component-System, що забезпечує ефективне використання кешу та мінімізацію витрат пам’яті.
Для багатокористувацького режиму застосовано підхід Deterministic Lockstep із реплікацією вводу, що зменшує мережеве навантаження. Також система демонструє високу масштабованість і надійність завдяки паралельним обчисленням незалежних підсистем і механізмам виявлення десинхронізації.

В.1 DOD (C++): структура даних 


struct Belts {
    std::vector<float> x, y, v;
    std::vector<bool> active;


    Belts(size_t capacity) {
        x.reserve(capacity);
        y.reserve(capacity);
        v.reserve(capacity);
        active.reserve(capacity);
    }


    void add(float px, float py, float vel) {
        x.push_back(px);
        y.push_back(py);
        v.push_back(vel);
        active.push_back(true);
    }


    size_t size() const { return x.size(); }
};


В.2 DOD (C++): обробка 
void update(Belts& b, float dt) {
    size_t n = b.size();


    for (size_t i = 0; i < n; ++i) {
        if (!b.active[i]) continue;
        b.x[i] += b.v[i] * dt;
    }
}
