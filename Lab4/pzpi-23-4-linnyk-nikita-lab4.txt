        Показати як можна масштабувати бекенд системи для роботи із великим навантаженням. Для цього, можна на вибір:
– масштабувати сервер горизонтально — багато копій сервера виконують однакові функції для різних користувачів;
– масштабувати сервер вертикально — різні мікросервіси виконують різні функції і масштабуються окремо одне від одного.
З технічної точки зору, масштабування можна виконати так:
– деплой сервера на Kubernetes кластер, разом із окремим load balancer-ом та базою даних. При цьому треба буде показати коректну роботу системи при
збільшенні чи зменшенні кількості подів сервера чи окремих мікросервісів;
– деплой в serverless середовище (AWS Lambda, Firebase Cloud Functions або аналогічне). Треба показати що система може зменшувати кількість активних інстансів до 0, збільшувати її зі збільшенням трафіку і все ще коректно функціонувати.
– ваш власний спосіб показати що система може масштабуватись.
На найвищий бал на цю роботу необхідно провести навантажувальне тестування за допомогою Gatling, JMeter, Locust чи іншого подібного інструмента і показати як зі збільшенням кількості серверів зростає кількість запитів на секунду яку витримує система.
3 ОПИС ВИКОНАНОЇ РОБОТИ
        Назва інструменту та версія: Anthropic Claude Sonnet 4.6
Мета використання: допомога у формуванні архітектури масштабованої серверної інфраструктури системи LangBang, опрацювання підходів до контейнеризації та горизонтального масштабування API-сервера, підготовка структури Kubernetes-конфігурацій, аналіз потенційних вузьких місць системи та підтримка під час проєктування сценаріїв навантажувального тестування.
Внесок автора: самостійне проєктування та реалізація серверної архітектури системи на Node.js / Express і PostgreSQL, контейнеризація застосунку за допомогою Docker, налаштування Kubernetes Deployment, Service, ConfigMap та Secret, реалізація endpoint перевірки стану /health для readinessProbe і livenessProbe, організація горизонтального масштабування API-сервера з різною кількістю реплік, проведення та аналіз навантажувального тестування через Locust, а також адаптація і доопрацювання запропонованих інструментом рішень відповідно до вимог лабораторної роботи.
3.1 Опис стратегії масштабування системи
Для системи LangBang було обрано стратегію горизонтального масштабування бекенда. Це означає, що замість запуску одного потужного сервера система використовує декілька однакових копій API-сервера, які паралельно обробляють запити користувачів.
Бекенд системи реалізовано на Node.js/Express і він взаємодіє з базою даних PostgreSQL через Prisma ORM. Оскільки основні дані системи зберігаються у спільній базі даних, API-сервери не повинні зберігати критичний стан локально в оперативній пам'яті. Це дає змогу запускати кілька однакових екземплярів сервера без порушення логіки роботи системи.
Загальну схему масштабування системи наведено на рисунку 3.1.
  

Рисунок 3.1 — Загальна схема горизонтального масштабування API-сервера


У межах лабораторної роботи масштабування демонструється зміною кількості реплік бекенда: 1, 2 та 4 pod-и. При збільшенні кількості реплік запити розподіляються між декількома екземплярами API через Kubernetes Service, а спільний стан зберігається у PostgreSQL.
Головним способом налаштування стенду було реалізовано локальний веб інтерфейс LangBang Lab 4 Setup. Він запускається на комп'ютері розробника і надає одну точку керування для Docker,  API Kubernetes, PostgreSQL, міграцій Prisma, масштабування запуску Locust. Завдяки цьому користувач працює переважно через UI: заповнює форму, натискає кнопки операцій і бачить журнал виконання.
3.2 Опис технічних рішень
Для забезпечення масштабування використовується контейнеризація за допомогою Docker. Бекенд упаковується в Docker image langbang-server:lab, який містить Node.js, код сервера, npm-залежності, згенерований Prisma Client та runtime-середовище для запуску TypeScript-коду. Це дає змогу запускати однакові копії сервера в ізольованому та повторюваному середовищі.
Для керування контейнерами використовується Kubernetes. Основними компонентами інфраструктури є:
– Deployment — описує бажану кількість запущених копій API-сервера;
– Pod — окремий запущений екземпляр бекенда;
– Service — стабільна мережева точка доступу, яка розподіляє запити між pod-ами;
– ConfigMap — зберігає некритичні налаштування середовища;
– Secret — зберігає конфіденційні дані, зокрема DATABASE_URL, JWT-секрети та API-ключі;
– PostgreSQL — спільна база даних для всіх екземплярів API;
– Job — одноразова задача для виконання міграцій Prisma та наповнення бази початковими даними.
Для перевірки працездатності сервера використовується endpoint перевірки стану:
GET /health
Kubernetes використовує цей endpoint для readinessProbe та livenessProbe. readinessProbe визначає, чи готовий pod приймати трафік, а livenessProbe дає змогу Kubernetes автоматично перезапустити pod у разі зависання або некоректної роботи.
Для зручного налаштування проєкту було реалізовано локальну веб панель:
scripts/lab4-web-setup.mjs
Файли інтерфейсу розміщено у каталозі:
scripts/lab4-web
Панель запускається з кореня репозиторію командою:
npm run lab4:web-setup
Після запуску користувач відкриває у браузері адресу:
http://localhost:8090
Саме вебпанель є основним способом виконання лабораторної роботи. Вона складається з декількох функціональних блоків:
– Deploy — форма повного розгортання стенду;
– Operations — окремі операції після розгортання;
– Import Data — імпорт даних у систему через API;
– Execution Log — журнал виконання команд і результатів.
У блоці Deploy користувач задає параметри середовища через UI. Можна обрати режим бази даних: локальний PostgreSQL у Kubernetes або зовнішню Neon PostgreSQL. Для локальної бази задаються POSTGRES_DB, POSTGRES_USER і POSTGRES_PASSWORD. Для Neon вводиться готовий DATABASE_URL. Також через форму задаються JWT_ACCESS_SECRET, JWT_REFRESH_SECRET, Cloudinary-параметри, початкова кількість API-реплік і додаткові змінні середовища.
Після заповнення форми користувач натискає кнопку Run full deploy. Вебпанель автоматично виконує послідовність дій: перевіряє наявність Docker і kubectl, за потреби збирає Docker image бекенда, застосовує ConfigMap, створює Kubernetes Secret, запускає PostgreSQL або використовує Neon, виконує Prisma migrations та seed через Kubernetes Job, деплоїть API-сервер і встановлює початкову кількість реплік.
Поточний стан операції відображається в Execution Log. У журналі видно запущені службові команди, відповіді Docker/Kubernetes, повідомлення про помилки або успішне завершення. Це важливо для демонстрації, тому що користувач керує процесом через UI, але водночас може бачити технічні деталі виконання.
Блок Operations використовується для керування вже розгорнутим стендом. У ньому доступні такі дії:
– Scale API — змінює кількість реплік API-сервера;
– Start port-forward — відкриває доступ до Kubernetes Service на localhost:5000;
– Check API — перевіряє endpoint-и /health та /api/courses;
– Install Locust — встановлює залежності для навантажувального тестування;
– Start Locust UI — запускає вебінтерфейс Locust на localhost:8089;
– Cleanup resources — видаляє створені Kubernetes-ресурси після завершення демонстрації.
Таким чином, масштабування виконується без ручного введення kubectl scale. Користувач вводить потрібну кількість реплік у полі API replicas і натискає Scale API. Після цього вебпанель змінює кількість pod-ів deployment/langbang-api, очікує завершення rollout і виводить список поточних pod-ів у журналі.
Важливою умовою масштабування є однакова конфігурація всіх API pod-ів. Усі екземпляри сервера повинні використовувати однакові JWT-секрети, однаковий DATABASE_URL і спільну базу даних. Завдяки цьому користувач може авторизуватися через один pod, а наступний запит може бути оброблений іншим pod-ом без втрати сесії.
3.3 Альтернативний консольний спосіб налаштування
Консольний спосіб можна використати як альтернативний варіант для випадків, коли потрібно виконати ті самі дії без браузера або детальніше проконтролювати окремі команди.
Для інтерактивного консольного запуску на Windows можна використати PowerShell-скрипт:
.\scripts\lab4-setup.ps1
У разі обмежень PowerShell Execution Policy скрипт можна запустити так:
powershell -ExecutionPolicy Bypass -File .\scripts\lab4-setup.ps1
Цей скрипт виконує подібні етапи: перевіряє Docker та kubectl, налаштовує NO_PROXY для Docker Desktop Kubernetes, збирає Docker image, застосовує Kubernetes ConfigMap, створює Secret, запускає PostgreSQL або використовує зовнішню базу, виконує міграції та seed, деплоїть API і змінює кількість реплік.
Також за потреби можна виконати окремі команди вручну. Наприклад, для масштабування API без UI використовується команда:
kubectl scale deployment/langbang-api --replicas=2
Для відкриття доступу до API через localhost використовується:
kubectl port-forward svc/langbang-api 5000:80
Однак у межах демонстрації ці дії виконувалися переважно через вебінтерфейс, оскільки він зменшує ймовірність помилки в послідовності команд і робить процес налаштування більш наочним.
3.4 Аналіз вузьких місць
Під час горизонтального масштабування API-серверів не всі компоненти системи масштабуються одночасно. Найімовірнішими вузькими місцями є база даних PostgreSQL, CPU pod-ів бекенда, кількість підключень Prisma до бази даних та ресурси локальної машини, на якій запускається Kubernetes-кластер.
При одній репліці основним обмеженням є CPU API-сервера. Один pod обробляє всі HTTP-запити, виконує валідацію, авторизацію, звернення до Prisma та формування JSON-відповідей. При зростанні навантаження CPU pod-а наближається до максимального значення, а час відповіді збільшується.
При двох репліках навантаження розподіляється між двома pod-ами, тому CPU кожного окремого pod-а використовується менше, а загальна кількість оброблених запитів за секунду може зростати. На цьому етапі система демонструє функціональну готовність до горизонтального масштабування.
При чотирьох репліках приріст продуктивності може ставати менш лінійним. Це пов'язано з тим, що всі pod-и продовжують звертатися до однієї бази даних PostgreSQL. У результаті зростає кількість одночасних підключень до бази, збільшується навантаження на SQL-запити, а база даних поступово стає основним вузьким місцем.
Окремим обмеженням локальної демонстрації є те, що тестування виконується через Docker Desktop Kubernetes, kubectl port-forward і Locust на тій самій машині. Через це результати можуть відображати не тільки продуктивність Express API, а й обмеження локального середовища.
Отже, горизонтальне масштабування API збільшує кількість екземплярів, які можуть приймати запити, але після певного рівня основними обмеженнями стають спільна база даних, мережевий шлях до сервісу та ресурси локального середовища. Для подальшого масштабування доцільно оптимізувати SQL-запити, додати connection pooling, кешування або масштабування бази даних.
3.5 Опис навантажувальних тестів
Для перевірки зв'язку між кількістю серверів і витримуваним навантаженням використовується Locust. Він імітує одночасну роботу багатьох користувачів, які виконують типові дії в системі LangBang.
Запуск тестування також інтегровано у вебпанель. Після розгортання API користувач у блоці Operations натискає Start port-forward, щоб відкрити доступ до Kubernetes Service за адресою:
http://localhost:5000
Далі користувач може натиснути Check API. Вебпанель виконує перевірку endpoint-ів /health та /api/courses і показує HTTP-статуси у журналі Execution Log. Це підтверджує, що API доступний перед запуском навантаження.
Після цього через кнопку Install Locust встановлюються залежності з load-tests/requirements.txt, а кнопка Start Locust UI запускає Locust. Вебінтерфейс Locust відкривається за адресою:
http://localhost:8089
У Locust задається однакова кількість віртуальних користувачів і однаковий spawn rate для всіх конфігурацій. Кількість реплік API між тестами змінюється через поле API replicas і кнопку Scale API у вебпанелі LangBang Lab 4 Setup.
Сценарій тестування включає такі запити:
– GET /api/courses — отримання списку курсів;
– GET /api/languages — отримання списку мов;
– POST /api/login — авторизація користувача;
– GET /api/leaderboard — отримання рейтингу користувачів після авторизації.
Тест запускався для трьох конфігурацій: 1, 2 та 4 репліки API-сервера. Для кожної конфігурації використовувалася однакова кількість віртуальних користувачів і однаковий сценарій запитів. Тести виконувалися протягом 2 хвилин.

4 ВИСНОВКИ
Таким чином, лабораторна робота демонструє, що бекенд системи може масштабуватися горизонтально за допомогою Kubernetes. Було створено Docker image бекенда, Kubernetes manifests для API, PostgreSQL, конфігурації, секретів та одноразової задачі міграцій. Основним способом налаштування стенду стала локальна вебпанель LangBang Lab 4 Setup, яка дає змогу через UI виконати повне розгортання, задати параметри середовища, обрати режим бази даних, масштабувати API, запустити port-forward, перевірити endpoint-и, встановити Locust, запустити Locust UI та очистити Kubernetes-ресурси після завершення роботи.
Консольний спосіб також залишено як альтернативний варіант: для цього доступний інтерактивний PowerShell-скрипт і ручні kubectl-команди. Проте основний сценарій демонстрації побудовано саме навколо веб інтерфейсу, оскільки він робить процес налаштування повторюваним, наочним і менш залежним від ручного введення команд у правильному порядку.
Збільшення кількості API-серверів дає змогу розподіляти навантаження між кількома pod-ами. У проведеному локальному тестуванні середній RPS майже не змінився при переході від 1 до 4 реплік, що свідчить про те, що вузьким місцем у цьому сценарії був не сам Express API, а спільні ресурси локального середовища: PostgreSQL, Docker Desktop, kubectl port-forward або генератор навантаження Locust.
Отже, масштабування API-рівня працює функціонально: система коректно запускається з різною кількістю pod-ів, обробляє запити без помилок і зберігає спільний стан у PostgreSQL. Для подальшого підвищення продуктивності доцільно оптимізувати роботу з базою даних, додати connection pooling, кешування або використовувати окреме production-середовище замість локального Docker Desktop.
