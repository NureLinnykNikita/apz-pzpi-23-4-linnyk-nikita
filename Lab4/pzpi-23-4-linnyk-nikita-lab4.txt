	2 ЗАВДАННЯ
        Показати як можна масштабувати бекенд системи для роботи із великим навантаженням. Для цього, можна на вибір:
– масштабувати сервер горизонтально — багато копій сервера виконують однакові функції для різних користувачів;
– масштабувати сервер вертикально — різні мікросервіси виконують різні функції і масштабуються окремо одне від одного.
З технічної точки зору, масштабування можна виконати так:
– деплой сервера на Kubernetes кластер, разом із окремим load balancer-ом та базою даних. При цьому треба буде показати коректну роботу системи при
збільшенні чи зменшенні кількості подів сервера чи окремих мікросервісів;
– деплой в serverless середовище (AWS Lambda, Firebase Cloud Functions або аналогічне). Треба показати що система може зменшувати кількість активних інстансів до 0, збільшувати її зі збільшенням трафіку і все ще коректно функціонувати.
– ваш власний спосіб показати що система може масштабуватись.
На найвищий бал на цю роботу необхідно провести навантажувальне тестування за допомогою Gatling, JMeter, Locust чи іншого подібного інструмента і показати як зі збільшенням кількості серверів зростає кількість запитів на секунду яку витримує система.
3 ОПИС ВИКОНАНОЇ РОБОТИ
        Назва інструменту та версія: Anthropic Claude Sonnet 4.6
Мета використання: допомога у формуванні архітектури масштабованої серверної інфраструктури системи LangBang, опрацювання підходів до контейнеризації та горизонтального масштабування API-сервера, підготовка структури Kubernetes-конфігурацій, аналіз потенційних вузьких місць системи та підтримка під час проєктування сценаріїв навантажувального тестування.
Внесок автора: самостійне проєктування та реалізація серверної архітектури системи на Node.js / Express і PostgreSQL, контейнеризація застосунку за допомогою Docker, налаштування Kubernetes Deployment, Service, ConfigMap та Secret, реалізація endpoint перевірки стану /health для readinessProbe і livenessProbe, організація горизонтального масштабування API-сервера з різною кількістю реплік, проведення та аналіз навантажувального тестування через Locust, а також адаптація і доопрацювання запропонованих інструментом рішень відповідно до вимог лабораторної роботи.
3.1 Опис стратегії масштабування системи
Для системи LangBang було обрано стратегію горизонтального масштабування бекенда. Це означає, що замість запуску одного потужного сервера система використовує декілька однакових копій API-сервера, які паралельно обробляють запити користувачів.
Бекенд системи реалізовано на Node.js/Express і він взаємодіє з базою даних PostgreSQL через Prisma ORM. Оскільки основні дані системи зберігаються у спільній базі даних, API-сервери не повинні зберігати критичний стан локально в оперативній пам’яті. Це дає змогу запускати кілька однакових екземплярів сервера без порушення логіки роботи системи.
Загальну схему масштабування системи наведено на рисунку 3.1.
  

Рисунок 3.1 — Загальна схема горизонтального масштабування API-сервера


У межах лабораторної роботи масштабування демонструється зміною кількості реплік бекенда: 1, 2 та 4 pod-и. При збільшенні кількості реплік очікується збільшення кількості запитів за секунду, які система може обробити.
3.2 Опис технічних рішень
Для забезпечення масштабування використовується контейнеризація за допомогою Docker. Бекенд упаковується в Docker image, який містить Node.js, код сервера, npm-залежності, згенерований Prisma client та зібраний TypeScript-код. Це дає змогу запускати однакові копії сервера в ізольованому та повторюваному середовищі.
Для керування контейнерами використовується Kubernetes. Основними компонентами інфраструктури є:
– Deployment — описує бажану кількість запущених копій API-сервера;
– Pod — окремий запущений екземпляр бекенда;
– Service — стабільна мережева точка доступу, яка розподіляє запити між pod-ами;
– ConfigMap — зберігає некритичні налаштування середовища;
– Secret — зберігає конфіденційні дані, зокрема DATABASE_URL, JWT-секрети та API-ключі;
– PostgreSQL — спільна база даних для всіх екземплярів API.
Для перевірки працездатності сервера використовується endpoint перевірки стану:
GET /health
Kubernetes може використовувати цей endpoint для readinessProbe та livenessProbe. readinessProbe визначає, чи готовий pod приймати трафік, а livenessProbe дає змогу Kubernetes автоматично перезапустити pod у разі зависання або некоректної роботи.
Важливою умовою масштабування є однакова конфігурація всіх API pod-ів. Усі екземпляри сервера повинні використовувати однакові JWT-секрети, однаковий DATABASE_URL і спільну базу даних. Завдяки цьому користувач може авторизуватися через один pod, а наступний запит може бути оброблений іншим pod-ом без втрати сесії.
3.3 Аналіз вузьких місць
Під час горизонтального масштабування API-серверів не всі компоненти системи масштабуються одночасно. Найімовірнішими вузькими місцями є база даних PostgreSQL, CPU pod-ів бекенда, кількість підключень Prisma до бази даних та ресурси локальної машини, на якій запускається Kubernetes-кластер.
При одній репліці основним обмеженням є CPU API-сервера. Один pod обробляє всі HTTP-запити, виконує валідацію, авторизацію, звернення до Prisma та формування JSON-відповідей. При зростанні навантаження CPU pod-а наближається до максимального значення, а час відповіді збільшується.
При двох репліках навантаження розподіляється між двома pod-ами, тому CPU кожного окремого pod-а використовується менше, а загальна кількість оброблених запитів за секунду зростає. На цьому етапі система масштабується майже лінійно.
При чотирьох репліках приріст продуктивності стає менш лінійним. Це пов’язано з тим, що всі pod-и продовжують звертатися до однієї бази даних PostgreSQL. У результаті зростає кількість одночасних підключень до бази, збільшується навантаження на SQL-запити, а база даних поступово стає основним вузьким місцем.
Отже, горизонтальне масштабування API збільшує пропускну здатність системи, але після певного рівня основним обмеженням стає база даних. Для подальшого масштабування доцільно оптимізувати SQL-запити, додати connection pooling, кешування або масштабування бази даних.
3.4 Опис навантажувальних тестів
Для перевірки зв’язку між кількістю серверів і витримуваним навантаженням використовується Locust. Він імітує одночасну роботу багатьох користувачів, які виконують типові дії в системі LangBang.
Сценарій тестування включає такі запити:
– GET /api/courses — отримання списку курсів;
– GET /api/languages — отримання списку мов;
– POST /api/login — авторизація користувача;
– GET /api/leaderboard — отримання рейтингу користувачів після авторизації.
Тест запускався для трьох конфігурацій: 1, 2 та 4 репліки API-сервера. Для кожної конфігурації використовувалася однакова кількість віртуальних користувачів і однаковий сценарій запитів. Тести виконувалися протягом 2 хвилин.

4 ВИСНОВКИ
Таким чином, лабораторна робота демонструє, що бекенд LangBang може масштабуватися горизонтально. Збільшення кількості API-серверів дає змогу обробляти більше запитів за секунду, але максимальна продуктивність системи залежить не лише від кількості серверів, а й від продуктивності спільних ресурсів, зокрема PostgreSQL.
